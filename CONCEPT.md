# Концепция OpenChat

## Проблема

Существующие мессенджеры (WhatsApp, Telegram, WeChat, Viber, ICQ) работают на закрытых корпоративных серверах. Пользователи зависят от одной компании, её политики и доступности.

## Решение

Децентрализованная система чатов:
- Любой может поднять свой сервер (бесплатно)
- Сервера могут быть частные (для себя/команды) или публичные
- Один клиент подключается к любому серверу
- Чаты локальны на сервере

**Аналоги**: IRC

## Цель проекта

Создать открытую экосистему для общения, где пользователь сам выбирает где хранить данные и кому доверять

## Устойчивость к блокировкам

### Проблема

Сервер может попасть в чёрный список и стать недоступным.

### Решение: Relay-сеть

1. **Whitelist relay** — сервер А явно разрешает серверу Б использовать себя как прокси
2. **Динамическая маршрутизация** — если прямой адрес недоступен, клиент ищет путь через доверенные ноды
3. **Multi-endpoint** — сервер публикует несколько способов подключения (прямой, через relay, через альтернативные порты)

### Преимущества

- Проще чем full mesh / DHT
- Доверие через явные связи, а не анонимную сеть
- Владелец сервера контролирует кто может его использовать как relay

## Идентификация и доверие

### Идентификатор пользователя

- **Email** — основной идентификатор (обязательный)
- Email идентифицирует пользователя в рамках чата

### Подключение к чату

**Сценарий 1: Сервер без доверенных**

1. Алиса хочет подключиться к чату на сервере А
2. Сервер А отправляет код/ссылку на email
3. Алиса подтверждает email → создаётся подключение (email + сервер + чат + ключ)
4. Приложение генерирует keypair, приватный ключ хранится локально

**Сценарий 2: Сервер с доверенными**

1. Алиса хочет подключиться к чату на сервере Б (у которого А в доверенных)
2. Сервер Б пытается запросить публичные ключи Алисы у сервера А
3. **Если А доступен и у Алисы есть подключения на А:**
   - Б получает публичные ключи
   - Алиса подписывает challenge в приложении приватным ключом
   - Б проверяет подпись локально
   - Если ок → подключение создано без подтверждения email
4. **Если А недоступен или нет подключений:**
   - Fallback на email-верификацию (как сценарий 1)

**Сценарий 3: Независимый сервер**

1. Алиса идёт на сервер В (нет доверенных)
2. Сервер В верифицирует через email
3. Создаётся подключение

### Identity Key (асимметричная криптография)

**Структура:**
- **Приватный ключ** — хранится в приложении на устройстве
- **Публичный ключ** — часть подключения (пользователь + сервер + чат + ключ)

**Принцип работы:**
1. При подключении к чату генерируется keypair
2. Для верификации на другом сервере — пользователь подписывает challenge приватным ключом
3. Доверенный сервер проверяет подпись по публичному ключу
4. После получения публичного ключа — верификация локальна, без повторных запросов

### Recovery и безопасность

- **Email recovery** — сброс через код на email
- **Отзыв подключения** — удаление с сервера
- **Уведомления** — при добавлении нового устройства в чат

### Модель доверия

```
┌─────────────────────────────────────────────────────────┐
│                      Сервер                          │
│  - Чаты                                                  │
│  - Подключения (email + ключ)                           │
│  - Список доверенных серверов                           │
└─────────────────────────────────────────────────────────┘
```

**Принципы:**

1. **Нет центрального узла** — каждый сервер автономен
2. **Доверенные сервера** — владелец сервера явно указывает каким серверам доверять

**Риски:**

- Плохие сервера могут образовать "теневую сеть"
- Решение: владелец сервера не добавляет подозрительные сервера в доверенные

### Чат

- Базовая единица системы
- Существует только на сервере
- На одном сервере — сколько угодно чатов
- Атрибуты:
  - Название
  - Описание
  - Иконка
  - Видимость: открытый или скрытый
  - Присоединение: разрешено или нет
  - Лимит пользователей
  - Белый список (email)
  - Чёрный список (email)

### Видимость

**Скрытый пользователь:**
- Не отображается в поиске на сервере
- Можно найти только по email

**Скрытый чат:**
- Не отображается в списке чатов сервера
- Можно найти только по chatId

### Добавление в чат

**Если присоединение разрешено:**
- Пользователь может вступить сам по chatId

**Если присоединение запрещено:**
- Пользователь не может вступить сам
- Нужно приглашение от участника чата

**Приглашение:**
- Публичный чат: любой участник может пригласить по email
- Защищённый чат: только владелец/модератор может пригласить по email

### Подключение

- Связь: **пользователь + сервер + чат + публичный ключ**
- У одного пользователя может быть несколько подключений (разные устройства)
- Каждое устройство = свой ключ

```
Пользователь (alice@gmail.com)
    │
    ├── Подключение 1: сервер А + чат X + ключ 1 (телефон)
    ├── Подключение 2: сервер А + чат X + ключ 2 (планшет)
    └── Подключение 3: сервер Б + чат Y + ключ 3 (телефон)
```

### Мульти-устройства в чате

**На сервере:**
- Каждый device = отдельное подключение
- В чате может быть 2, 3 и более подключений одного email
- Это нормально, сервер хранит все как есть

**На клиенте (приложение):**
- Группирует подключения по email
- UI: "алиса (2)" или "алиса (3)" — показывает количество устройств

## Доставка сообщений

### Принцип

Сообщения отправляются в чат, не конкретному пользователю. Чат локален на сервере.

### Схема

1. Пользователь подключается к чату на сервере (подключение = email + сервер + чат + ключ)
2. Сообщение отправляется в чат на сервере
3. Сервер рассылает сообщение всем подключениям в чате
4. Каждое сообщение имеет уникальный ID

### Преимущества

- **Простота** — чат локален, нет синхронизации между серверами

## Шифрование

### Типы чатов

**Публичный чат:**
- Сервер генерирует ключ чата
- Сервер может расшифровать сообщения
- Если не доверяешь серверу — поднимаешь свой

**Защищённый чат (E2E):**
- Первый подключившийся (владелец) генерирует ключ чата
- Сервер не знает ключ
- Настоящее E2E шифрование

### Общий ключ чата

1. При создании чата сервер генерирует **симметричный ключ чата**
2. Ключ чата шифруется публичным ключом каждого подключения
3. При подключении участник получает зашифрованный ключ чата
4. Сообщение шифруется один раз ключом чата
5. Сервер рассылает один шифротекст всем

### Схема

**Публичный чат (сервер генерирует ключ):**
```
Чат создаётся → сервер генерирует ключ K
Ключ K шифруется публичными ключами подключений
```

**Защищённый чат (E2E, владелец генерирует ключ):**
```
Чат создаётся → первый подключившийся генерирует ключ K
Ключ K шифруется публичными ключами подключений
Сервер не знает K
```

### Вход нового участника

**Публичный чат:**
1. Новый участник генерирует keypair
2. Сервер шифрует ключ чата публичным ключом новичка
3. Новичок получает ключ

**Защищённый чат (E2E):**
1. Новый участник генерирует keypair
2. Владелец (или любой онлайн-участник с ключом) шифрует ключ чата публичным ключом новичка
3. Если все оффлайн — новичок ждёт появления владельца

### Преимущества

- Одно шифрование на сообщение
- Нагрузка не растёт с количеством участников
- Защита от перехвата в сети (MITM)
- E2E опционально (через ролевую систему)

## Ролевая модель

### Виды действий

**Сообщения:**
- Читать сообщения
- Писать сообщения
- Редактировать и удалять свои сообщения
- Редактировать и удалять сообщения других участников

**Оценки:**
- Оценивать сообщения
- Редактировать и удалять свои оценки
- Редактировать и удалять оценки других участников

### Роли

**Тип чата выбирается при создании: публичный или защищённый.**

**Публичный чат:**
- Все участники равны (роль Участник)
- Ключ чата генерирует сервер

**Защищённый чат:**
- Есть роли: Владелец, Модератор, Участник, Наблюдатель
- Владельцы назначают роли участникам
- Ключ чата генерирует первый подключившийся (владелец) → E2E шифрование

### Роли и права

**Владелец/Суперадмин:**
- Полный контроль над чатом
- Управляет списком владельцев
- Все права Модератора

**Модератор:**
- Читать сообщения
- Писать сообщения
- Редактировать и удалять свои сообщения
- Редактировать и удалять сообщения других участников
- Оценивать сообщения
- Редактировать и удалять свои оценки
- Редактировать и удалять оценки других участников

**Участник:**
- Читать сообщения
- Писать сообщения
- Редактировать и удалять свои сообщения
- Оценивать сообщения
- Редактировать и удалять свои оценки

**Наблюдатель:**
- Читать сообщения
- Оценивать сообщения
- Редактировать и удалять свои оценки

## История сообщений

### Хранение

**Сервер** — основное хранилище:
- Ограничение по настройке чата
- В защищённом чате хранит шифротексты (без ключа не читаемы)

**Клиент** — кэш:
- Для оффлайн доступа
- Для быстрой загрузки

### Логика

- Новое подключение → клиент скачивает историю с сервера
- При выходе из чата → кэш клиента очищается

### Ограничения

- По времени (хранить N дней)
- По количеству (хранить последние N сообщений)
- Без ограничений (бесконечно)

## Оффлайн доставка

### Принцип

Сервер хранит сообщения до получения клиентом.

### Очередь сообщений

- Каждое сообщение имеет статус: отправлено → доставлено → прочитано
- При подключении клиент скачивает непрочитанное из очереди
- Учитывается в лимите истории чата

### Статусы сообщений

- **Отправлено** — сервер получил
- **Доставлено** — клиент получил
- **Прочитано** — клиент отобразил пользователю
